# Privacy Principles

## Purpose

Accountability Atlas should collect as little personal data as possible and avoid exposing users, contributors, witnesses, or affected communities.

---

## MVP privacy posture

For the first MVP:

- no user accounts;
- no sensitive submissions;
- no witness uploads;
- no behavioral tracking;
- no advertising pixels;
- no invasive analytics;
- no unnecessary cookies;
- no private messaging;
- no public user profiles.

---

## Data minimization

Only collect data when it is necessary for a clear user benefit or operational requirement.

Ask before collecting:

1. Why do we need this?
2. Can we avoid collecting it?
3. Can we anonymize it?
4. Can we store it for less time?
5. Who can access it?
6. What harm happens if it leaks?

---

## Analytics

If analytics are used, prefer privacy-first tools.

Analytics should avoid:

- individual user tracking;
- cross-site tracking;
- advertising identifiers;
- fingerprinting;
- unnecessary IP storage;
- cookies where possible.

Public dashboards should avoid showing data that could expose vulnerable users in small countries or dangerous contexts.

---

## Contact forms

If contact forms are used:

- collect only required fields;
- explain why fields are needed;
- avoid sensitive content;
- use spam protection that does not over-track users;
- define retention period;
- protect access.

---

## Newsletter

If a newsletter is added:

- use explicit opt-in;
- provide unsubscribe;
- avoid selling/sharing lists;
- do not mix donor lists, action lists, and contributor lists without consent;
- state data use clearly.

---

## Action tracking

Action counters can be useful, but should be privacy-preserving.

Prefer:

- aggregated counts;
- country-level totals only when safe;
- no public user identities;
- no ranking of individuals;
- no tracking of exact message content unless user consents.

---

## Sensitive future workflows

Witness submissions, encrypted uploads, or source protection workflows require separate privacy impact assessment before development.

Required before sensitive workflows:

- threat model;
- data map;
- retention policy;
- access control plan;
- encryption plan;
- deletion process;
- legal review;
- emergency response process.

---

## GDPR orientation

The project should be designed to respect EU privacy principles:

- lawfulness, fairness, and transparency;
- purpose limitation;
- data minimization;
- accuracy;
- storage limitation;
- integrity and confidentiality;
- accountability.

This document is not legal advice. Formal legal review is needed before launch if personal data is processed.
